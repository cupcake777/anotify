# anotify desktop — implementation notes

The desktop app is a Tauri 2 application with **two windows**:

- **`main`** (`src/index.html`) — the dashboard: live status, a notification
  inbox with inline Accept/Deny for approvals, and settings.
- **`toasts`** (`src/toast.html`) — a transparent, always-on-top overlay that
  renders the animated pixel-bird toasts in the bottom-right corner.

The Rust backend (`src-tauri/src/`) owns the WebSocket connection to the relay
and forwards every notification to both windows.

## Layout


## Data flow

```
relay  ──WS──▶  ws.rs (connect / dedup / history baseline+replay)
                  │  annotate kind
                  ├─ emit_to("toasts", "notification", n)   → bird toast
                  └─ emit_to("main",   "notification", n)   → inbox card
                                                  (+ "history", "connection")
toast / dashboard  ──invoke("respond_approval")──▶  main.rs
                                                      │ POST /api/approval/respond
                                                      ▼
                                                    relay → agent
```

## Commands (Rust → JS via `invoke`)

| Command | Purpose |
|---|---|
| `get_config` | Read config (token returned masked as `__SET__`). |
| `update_config` | Save server/token/dnd/autostart; forces an immediate reconnect. |
| `verify_connection` | Current connected state. |
| `reconnect` | Drop the socket and reconnect now. |
| `get_notifications` | In-memory history (bounded to 200). |
| `clear_notifications` | Empty the history buffer. |
| `respond_approval` | Relay an Accept/Deny/Acknowledge back to the agent. |

## Approval flow (end to end)

Approvals work with an **outbound-only** round-trip, so they function from a
locked-down remote host (no inbound reachability needed):

1. An agent runs `anotify approve "Deploy to prod?" --action … --target …`.
   The CLI POSTs an approval notification (with an `approval_id`) and then
   long-polls `GET /api/approval/wait/{id}`.
2. The relay broadcasts it; the desktop shows an approval toast (and an inbox
   card) with **Approve / Deny**.
3. The user clicks → `respond_approval` → relay records the decision (and
   forwards to a local `callback_url` if one was set).
4. The CLI's long-poll returns the decision and exits `0` (approved) / `1`
   (denied) / `2` (timeout), so scripts can gate on it:

   ```bash
   if anotify approve "Deploy to prod?"; then ./deploy.sh; fi
   ```

The Python desktop client (`anotify-client` / `anotify-mac`) resolves approvals
too, via a native dialog (macOS `osascript`, tkinter fallback).

## Run (requires the Rust + Tauri toolchain)

```bash
cd desktop
pnpm install          # or npm install
pnpm tauri dev        # cargo + tauri must be installed
```

Point it at your relay (Settings tab, or `~/.anotify.json` /
`ANOTIFY_SERVER` + `ANOTIFY_TOKEN`), then from any machine:

```bash
anotify send "build done" --title "HPC" --source slurm@hpc-login \
  --script smr_sweep.sh --priority medium
# or request a decision:
anotify approve "Merge PR #42?" --agent claude-code
```

A bird toast slides in at the bottom-right; the dashboard inbox updates live.

## On-device checklist (can't be verified headless)

- [ ] Toast window is transparent (no white box) on your OS.
- [ ] Bird is crisp (eyes are two dots, not blurry) — frames are sliced at
      runtime from the bundled strips.
- [ ] Sprite plays its enter animation, then settles.
- [ ] **Critical** stays until Acknowledge/× and re-pings every 60s.
- [ ] **Approval** shows Approve/Deny, does not auto-dismiss, and the decision
      reaches the waiting `anotify approve` (exit code matches).
- [ ] Dashboard inbox shows the same approval and resolving it in either place
      is reflected.
- [ ] When the last toast clears, the overlay window hides.

## Known limitations (candidates for next pass)

- The overlay captures clicks within its corner box while visible (it hides
  when empty). True per-toast click-through is a refinement.
- Dashboard Settings → overlay live push (`toast-settings` event) is stubbed in
  the overlay but not yet driven by the dashboard UI.
- Tray icon / menu for the Tauri app is not wired yet (the Python client and
  macOS menu-bar app have one); the dashboard window is the control surface.
